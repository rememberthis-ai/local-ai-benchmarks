# whisper.cpp Metal on Polaris dGPU: SIGABRT on long audio

Discovered while running `transcription-bench/sona_bench.sh` on 2026-05-17. The same hardware that works fine for short clips reliably crashes whisper.cpp/ggml-metal when audio length grows past a few minutes.

## Reproducer

- **Hardware**: 2018 MacBook Pro 15" — Intel Core i9-8950HK + AMD Radeon Pro 560X (Polaris, 4 GB dedicated VRAM)
- **Software**: sona v0.1.4 (whisper.cpp wrapper, bundled in Remember This v0.10) + `ggml-large-v3-turbo.bin`
- **Audio**: 10-min slice of the LibriVox Sherlock Holmes ch.1 clip (`audio/holmes_clip600.wav` in the bench repo, generated by `ffmpeg -ss 0 -t 600 -ar 16000 -ac 1` from `adventureholmes_01_doyle_64kb.mp3`)
- **Command**:
  ```bash
  sona serve --port 5530          # NO --no-gpu → Metal on Polaris
  # then POST /v1/models/load + POST /v1/audio/transcriptions with n_threads=4
  ```

## Symptom

After ~120-180 s of CPU-side activity (audio buffering, encoder warmup), sona aborts:

```
/Users/runner/work/sona/sona/whisper-src/ggml/src/ggml-metal/ggml-metal-device.m:1561:
GGML_ASSERT(buf_src) failed
WARNING: Using native backtrace. Set GGML_BACKTRACE_LLDB for more info.
0   sona  ggml_print_backtrace + 267
1   sona  ggml_abort + 261
2   sona  ggml_metal_buffer_set_tensor + 527
3   sona  ggml_backend_sched_graph_compute_async + 1300
...
SIGABRT: abort
PC=0x7ff80afab846 m=8 sigcode=0
signal arrived during cgo execution
```

Same backtrace on retry. Deterministic, not transient.

## What works vs what doesn't

| Setup | Holmes 60 s | Holmes 10 min |
|---|---|---|
| `sona serve` (Metal default → Polaris dGPU) | ✅ 99.89 s wall | ❌ SIGABRT @ ~3 min |
| `sona serve --no-gpu` (CPU only) | ✅ 47.78 s wall | ✅ 578.99 s wall |
| Apple Silicon `sona serve` (Metal on UMA) | ✅ 1.39 s wall | ✅ 16.70 s wall |

So:
- Polaris Metal works for short clips
- Polaris Metal reliably fails for ≥10 min clips with large-v3-turbo
- Apple Silicon Metal works at every length tested

Crash threshold not pinpointed — somewhere between 1 and 10 minutes; not bench-mapped.

## Why this matters for product

The RT/MT app currently exposes a "Force CPU transcription" toggle in Settings (Settings → Audio Transcription on the native app). On Intel + Polaris, this toggle is **not just a perf trade-off; it's a stability requirement** for any audio longer than a few minutes. The shipping default for `cpu_only_transcription` on Intel should be **on**, not off — opposite of what intuition ("Metal is for offload, faster") would suggest.

For My Transcriber specifically, which is purpose-built for transcribing meetings (typically 30-60 min), the Metal path on Intel + Polaris would crash on essentially every user file. Force CPU.
